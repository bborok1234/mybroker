# MyBroker Surfaces

MyBroker has four separate surfaces. They must not collapse into one dashboard.

## 1. Flyhigh Operator Dashboard

Purpose: project implementation progress.

Shows issues, pull requests, direction reviews, merge gates, blockers, validation, live repo state, and next actions. It may link to MyBroker artifacts, but it must not render product market maps or user action candidates inline.

## 2. MyBroker Ops/Artifact Dashboard

Purpose: artifact trust and local run inspection.

Shows report counts, source coverage, freshness, validation, generated artifact paths, data quality, and blockers. It is for builders/operators, not beginner users.

Must not be treated as the product UX.

For the autonomous daily research loop it may show topic coverage, latest run status, source freshness, collection gaps, topic memory health, and artifact paths. It still must not render the beginner market map or action candidates as the main product surface.

## 3. MyBroker Product Brief

Purpose: beginner-facing market understanding.

Generated by `mybroker brief`, this is the MiroFish-inspired user surface. It shows market/narrative maps, key themes, three scenario paths, persona viewpoints, beginner explanations, next inspection questions, and research-only action candidates.

Must not show adapter IDs, schema names, raw validation logs, Flyhigh gates, or implementation details.

## 4. Machine Artifacts

Purpose: reproducibility and audit.

Machine-readable artifacts include:

- `public_evidence_catalog.v1`
- `topic_config.v1`
- `daily_research_plan.v1`
- `topic_memory.v1`
- `scenario_report.v1`
- `market_verdict.v1`
- `research_report.v1`

These are inputs for validators, dashboards, product briefs, and Flyhigh evidence. They are not user-facing screens.
